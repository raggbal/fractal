# Release v0.195.714

**リリース日:** 2026-04-19
**バージョン種別:** patch (refactor only, no behavior change)
**対応要件:** v12 (OL-30) の内部構造改善

## 概要

v0.195.712 / v0.195.713 で追加した D&D ファイル import の**内部構造**を DRY 原則に沿って整理したリファクタリリース。ユーザー機能は一切変更なし。

## 変更内容

### Changed
- `src/shared/drop-import.ts`:
  - `saveImageBuffer(buffer, extHint, ctx)` と `saveImageFromDataUrl(dataUrl, ctx)` を抽出。Finder 経路 (`processDropFilesImport`) と Explorer 経路 (`processDropVscodeUrisImport`) で重複していた `image_${Date.now()}_${Math.random()...}.${ext}` 生成 + `mkdir` + `writeFileSync` + `path.relative` の 10 行程度のコードブロックを 1 関数に集約
  - `createDropImportHandler<P>(processor, deps)` factory を追加。`resolveDirs` / `postMessage` / `getDisplayUri` / `onFailed` を closure として受け取り、統一した handler を返す
- `src/outlinerProvider.ts`:
  - `case 'dropFilesImport':` と `case 'dropVscodeUrisImport':` (計 59 行) を factory 経由の one-liner (計 6 行) に書き換え
  - message handler セットアップ前に `handleFinderDrop` / `handleExplorerDrop` をインスタンス化 (共通 `dropHandlerDeps` を使い回し)
- `src/notesEditorProvider.ts`:
  - `NotesPlatformActions.dropFilesImport` / `dropVscodeUrisImport` (計 59 行) を factory 経由の one-liner に書き換え
  - `makeNotesDropHandler(senderRef, processor)` ヘルパーで senderRef を closure で受け取る形に整理
  - `fileManager.getCurrentFilePath()` null guard は両ハンドラで明示的に保持

### 動機
v0.195.713 で発生した「Notes mode の `dropVscodeUrisImport` ハンドラ抜け」バグの**根本原因は構造的な重複**にあった。同一骨格の handler body を Finder/Explorer × standalone/Notes の 4 箇所に手書きしていたため、片方を追加するだけで他方を忘れるリスクが構造的に存在した。factory 化により「新経路/新プラットフォーム追加時に boilerplate をコピペする必要がない」状態に改善。
