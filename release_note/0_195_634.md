# 0.195.634

## 概要
0.195.624 までの retro で積み残していた技術債と、その過程で発見した cross-outliner paste の根本的な欠陥をまとめて解消した。identity ベースの同一性判定に統一し、host 側の .md/画像操作 API も 1 本化した。

## 対象バージョンでまとめて直したこと (0.195.625 〜 0.195.634)
1. `extractMarkdownImagePaths` の重複を `src/shared/markdown-image-utils.ts` に集約
2. 相対パス文字列 (`./pages`) での cross-file 判定を撤廃し、`outFileKey` (.out ファイルの絶対パス) による identity 判定に統一
3. host 側 `copyPageFile`/`copyPageFileCross` の 2 本立てを `copyPageFileCross` 1 本に統合
4. cmd+c 時に画像ファイルを新 filename で実体コピーし、.md 本文の画像参照も rewrite
5. cmd+x 時に cross-folder なら .md と画像を物理移動
6. cross-webview (別 note 間) の cmd+x で cut 情報が失われていたバグ修正
7. Webview 初期化時に `outFileKey` が渡されず identity 判定が常に false になっていたバグ修正

## 課題

### 課題 A: extractMarkdownImagePaths の重複 (retro 教訓③)
`src/outlinerProvider.ts` と `src/shared/notes-message-handler.ts` に**完全に同一**の `extractMarkdownImagePaths` 関数が丸コピーされていた。0.195.624 の hotfix で「後で共通化」と先送りしていた技術債。

### 課題 B: 相対パス文字列での同一性判定 (retro 教訓①)
`isCrossFile = forceCrossFile || (srcPageDir && srcPageDir !== pageDir)` の後半の判定根拠が相対パス文字列で、別 outliner でも `./pages` が偶然一致して非 cross 経路に落ちていた。retro で「直した」と宣言したものの実際には `forceCrossFile` を足しただけで、別 outliner 間の paste は壊れたままだった。

### 課題 C: 同じ note / 別 outliner 間の page node copy/paste 不具合
同じ note フォルダ内の 2 つの .out ファイル間で page ノードを copy/paste / cut/paste しても .md が複製/移動されない。retro 0.195.623 時点から存在していた既存バグ。

### 課題 D: 画像の複製挙動がユーザー期待と乖離
cmd+c で page ノードを複製すると、同じ outliner 内では画像ファイルが「参照共有」になり、元を消すと複製先でも消える状態。ユーザーは常に独立した画像ファイルを期待していた。

### 課題 E: cut 情報が cross-webview paste で失われる
別 note の webview に paste する際、HTML clipboard metadata に `isCut` フラグが含まれておらず、受け側が `isCutPaste = false` をハードコードしていた。そのため cmd+x が事実上 cmd+c として処理され、元 note の .md/画像が削除されなかった。

### 課題 F: Webview 初期化時に outFileKey が欠落
Notes webview は初回ロードで `Outliner.init(data)` を直接呼び、`updateData` メッセージ経由ではない。`outFileKey` が渡されず `currentOutFileKey = null` の状態で paste すると、`isCrossFile` が常に false になり cross 経路が動作しなかった。

## 原因

### 原因 A
hotfix が重なり、共通化するタイミングを逃した。

### 原因 B
0.195.623 の修正は HTML metadata 経由のペーストに `forceCrossFile = true` を追加しただけで、intClip パス (同一 webview 内での file 切替後のペースト) には効果がなかった。旧条件 `srcPageDir !== pageDir` を残したまま併用していたため、真の原因 (相対パス比較) が温存されていた。

### 原因 C
[notesEditorProvider](src/notesEditorProvider.ts) は 1 つの webview で複数の .out ファイルを切り替える設計。webview の JS クロージャ変数 `internalClipboard` は file 切替で保持される一方、host 側では `copyPageFile` (同一 dir 前提) が呼ばれ、B.out の pagesDir に存在しない .md を探して**無言で失敗**していた。

### 原因 D
旧実装は「同一 file 内は `clipNode.images.slice()` で path 流用」としていた。これは画像参照を共有するだけで、独立した複製ではない。

### 原因 E
`writeClipboardWithHtml` の metadata JSON に `isCut` を含めていなかった。cross-webview paste では webview の内部変数 `internalClipboard` が参照できないため、metadata だけが情報源。

### 原因 F
`Outliner.init(data)` シグネチャが `data` のみを受け取っていた。notesWebviewContent.ts / outlinerWebviewContent.ts は base64 埋め込みの .out JSON を渡すが、親の .out ファイルパスは渡していなかった。`updateData` メッセージは編集同期や file 切替の時だけ送られるので、初回ロード直後は `currentOutFileKey = null` のままだった。

## 対策

### 設計

#### 1. 共通化 (課題 A)
`src/shared/markdown-image-utils.ts` を新規作成。正規表現をモジュールスコープ定数 `MD_IMG_RE` / `HTML_IMG_RE` として保持する純関数 + ローカル `push` クロージャで実装。

#### 2. identity ベース判定 (課題 B, C)
.out ファイルの**絶対パス**を唯一の identity とする:
- `OutlinerClipboardStore` に `sourceOutDir` フィールドを追加
- `updateData` protocol に `outFileKey` を追加
- `Outliner.init(data, outFileKey)` の第 2 引数で初回ロード時も注入
- webview の closure 変数 `currentOutFileKey` で保持
- `internalClipboard` / HTML clipboard metadata に `sourceOutFileKey` を埋め込み
- `pasteNodesFromText` の `isCrossFile` 計算を `clipSourceKey !== currentOutFileKey` に変更
- 相対パス文字列 `sourcePageDir` は完全撤廃

#### 3. host API 統合 (課題 C)
`copyPageFile` (同一 dir 前提) を 2 ファイルから削除し、`copyPageFileCross` に一本化。`copyPageFileCross` は常に `OutlinerClipboardStore` の絶対パスから source を解決するため、同一 file でも cross-file でも同じコードパスで動く。

#### 4. 共通 paste asset handler (課題 C, D)
`src/shared/paste-asset-handler.ts` を新規作成:
- `copyPageAssets`: .md を新 pageId でコピー + 画像を新 filename で実体コピー + .md 本文の画像参照を rewrite
- `movePageAssets`: .md を物理移動 (src !== dest) + 画像を物理移動
- `copyImageAssets` / `moveImageAssets`: 非 isPage ノードの images 用

新 filename 命名規則: `copy-<newPageId>-<originalFilename>` で衝突回避。

#### 5. postback で newNode.images を更新 (課題 D)
host で画像 path が rewrite されるため、webview 側では新 path を知らない。host が `updateNodeImages` メッセージで新 path を webview に返し、webview が該当 node の images を書き換えて re-render + sync する非同期設計。

#### 6. cut 情報の保存 (課題 E)
`writeClipboardWithHtml(plainText, nodesData, isCut)` に第 3 引数追加。HTML clipboard metadata JSON に `isCut: boolean` を埋め込み、受け側 crossMeta ブランチで `isCutPaste = !!crossMeta.isCut` に変更。

#### 7. 初期化時の outFileKey 注入 (課題 F)
`getNotesWebviewContent` / `getOutlinerWebviewContent` で `Outliner.init(initialData, ${currentFilePath})` のように outFileKey を注入。`outlinerProvider.ts` は `document.uri.fsPath` を渡す。

#### isCrossFile の最終形
```js
var isCrossFile = !!clipSourceKey && clipSourceKey !== currentOutFileKey;
```

#### 振る舞いまとめ
| シナリオ | cmd+c | cmd+x |
|---|---|---|
| 同じ outliner (same file) | 新 pageId で .md 複製 + 新 filename で画像コピー + 本文 rewrite | no-op (pageId/images 不変) |
| 同じ note / 別 outliner (same pagesDir) | 同上 | .md の pageId 保持、物理 move は pagesDir 同じなので no-op |
| 別 note (different pagesDir) | 同上 | .md と画像を物理移動 (src copy → unlink) |
