# Release v0.195.646

**リリース日:** 2026-04-12
**バージョン種別:** minor (新機能追加 + バグ修正)
**対応要件:** requirement v7 → v7.1 → v7.2 / design v7 → v7.1 → v7.2 → v7.3
**ブランチ:** feature/delete-file-cleanup-v7 (705b3cb ベース)

## 概要

Notes editor の未参照ファイル (孤立 .md + 孤立画像) を手動でクリーンアップできる機能を追加。
削除時の物理ファイル自動削除は Undo/Redo との整合性問題のため採用せず、ユーザーが意図的にクリーンアップコマンドを実行する方式を採用した。
併せて、Remove Page 時に .md が物理削除される main の既存バグも修正し、Undo で完全復元可能にした。

## 変更内容

### 新機能

- **FR-5: 手動クリーンアップコマンド** — `Fractal: Clean Unused Files in All Notes` (コマンドパレット)
  - 登録済み全 note をスキャンし、未参照 .md + 孤立画像を 2-pass で検出
  - QuickPick で候補を note ごとにセパレータ付き表示、全選択/全解除ボタン
  - 確定後に vscode.workspace.fs.delete({ useTrash: true }) でゴミ箱移動
  - 結果レポート (件数 + 容量) を通知
  - Progress UI (vscode.window.withProgress) でスキャン進捗表示、キャンセル可能

- **FR-9: 自ノート限定クリーンアップ** — `Fractal: Clean Unused Files in Current Note` (コマンドパレット)
  - 現在開いている note のみをスキャン (全 note モードとは別に)
  - Tools タブの Clean Notes セクション内にもボタンあり

- **FR-8: Tools タブ** — Notes editor の既存 S3 タブを "Tools" / "ツール" にリネーム
  - S3 Sync セクション (既存機能そのまま) + Clean Notes セクション (新規) の 2 セクション構成
  - Clean Notes セクションに「自ノート」「全ノート」の 2 ボタン

- **FR-3: 起動時マイグレーション** — .out ファイルの `schemaVersion` を管理
  - schemaVersion 未設定の .out を開くと、重複画像 (複数ノードが同じ画像パスを共有) を自動検出 → 物理複製して 1画像=1所有者に修復
  - 冪等 (2回実行しても変化なし)、失敗安全 (schemaVersion 未更新のまま)
  - 新規 .out は createFile() で schemaVersion: 2 を最初から設定

### バグ修正

- **FR-4: handleRemovePage の物理削除停止** — main (v0.195.641) で Right-click → "Remove Page" すると .md ファイルが物理削除され、Undo しても復活しなかった既存バグを修正。.md をオーファンとして残し、cleanup コマンドで掃除する方式に変更。Undo/Redo で完全復元可能
- **FR-1: notes-file-manager.deleteFile の useTrash 化** — Notes panel の outliner 削除時、`fs.unlinkSync` / `fs.rmSync` → `vscode.workspace.fs.delete({ useTrash: true })` に変更。ゴミ箱から復元可能に
- **cleanup-core.ts 画像パス解決バグ** — buildLiveSetPass1 が node.images[] を pageDir 基準で resolve していたが、実際の保存は outDir 基準。alive 画像が orphan 誤判定されて削除されるバグを修正
- **switchTab バグ** — Tools タブをクリックしても何も表示されない問題を修正 (filePanelContentS3 → filePanelContentTools)

### 非機能改善

- **NFR-1: 安全性** — 全削除は useTrash:true で OS ゴミ箱移動。即時削除 API (unlinkSync/rmSync) は src/ から撤去 (paste-asset-handler.ts と notes-s3-sync.ts は例外)
- **NFR-2: Undo/Redo 整合性** — エディタ内操作で物理ファイルを触らないため、Undo で完全復元
- **NFR-8: パストラバーサル防御** — path-safety.ts の safeResolveUnderDir() でファイルパス検証
- **NFR-9: S3 機能退行なし** — Tools タブ化後も S3 Sync 機能は全て正常動作

### 主要な実装判断

1. **削除時カスケード vs 手動クリーンアップ**: v6 で「削除時にファイルもゴミ箱移動」を実装したが、Undo/Redo との整合性が取れず断念 (v6 の教訓)。v7 で手動クリーンアップ方式に転換
2. **1画像=1所有者**: main の既存 `copyImageAssets` / `copyPageAssets` が既に同一 outliner 内 copy でも物理複製していることを発見 (v7.1)。新規実装 (`duplicateImages`) は不要だった
3. **2-pass cleanup**: 孤立 .md を先に特定し (Pass 1)、その .md からの画像参照を「alive でない」とみなして孤立画像を検出 (Pass 2)。1回の cleanup で transitive closure を実現
4. **outDir 基準のパス解決**: node.images[] は `path.relative(outDir, destPath)` で保存されている。pageDirAbs 基準で resolve すると誤ったパスになる (v7.3 バグ修正)
5. **NotesFolderProvider.getFolders()**: VSCode Activity Bar の Notes Explorer に登録された全 note フォルダを取得して全 note 一気スキャン

## 既知の制限事項・残課題

- **paste-asset-handler.ts の fs.unlinkSync 3 箇所**: cut-paste cross-file の move semantics。実害ゼロだが将来 `vscode.workspace.fs.rename()` に置き換え推奨
- **notes-s3-sync.ts の fs.unlinkSync/rmSync**: S3 同期の即時削除。リモートから再取得可能なセマンティクスで正当
- **Standalone outliner の削除対応**: VSCode Explorer から .out を直接 delete した場合の関連ファイル cleanup はスコープ外
- **notes-daily.spec.ts のフレーキーテスト**: 並列実行時のみ typing race condition で fail。v7 とは無関係の既存問題
- **fractal-md skill の取り込み精度**: aws note で 13,098 件の orphan ファイルが検出される。cleanup 機能の正常動作であり、取り込み時の紐付け失敗 (約 48%) は fractal-md skill 側の問題
